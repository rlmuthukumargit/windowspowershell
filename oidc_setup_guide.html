Phase 2: Direct OIDC Setup (Concept 1)

In this pattern, the Azure DevOps pipeline assumes the role directly.

1

Create Identity Provider

1. Open IAM Console > Identity providers > Add provider.

2. Provider URL: https://vstoken.dev.azure.com/{ORG_ID}

3. Audience: api://AzureADTokenExchange

2

Create Target Role

Create a Role with Web Identity trust relationship. Attach permissions required for the deployment.

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": { "Federated": "arn:aws:iam::123456789012:oidc-provider/vstoken.dev.azure.com/{ORG_ID}" },
      "Action": "sts:AssumeRoleWithWebIdentity",
      "Condition": {
        "StringEquals": { "vstoken.dev.azure.com/{ORG_ID}:aud": "api://AzureADTokenExchange" }
      }
    }
  ]
}
Enterprise Pattern

Phase 3: AWS STS Token Broker (Concept 2)

The Token Broker pattern natively exchanges the OIDC identity for a secondary role via the Azure DevOps Service Connection UI (No PowerShell script required).

Configuration: The Service Connection Broker

This implementation allows you to separate the OIDC Identity Role (Role A) from the Deployment Task Role (Role B). Role A acts as the "identity broker" that assumes Role B.

A

Role B (Target) Trust Policy

Role B must trust Role A (the Role being assumed by the service connection).

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": { "AWS": "arn:aws:iam::123456789012:role/FederationIdentityRole_A" },
      "Action": "sts:AssumeRole"
    }
  ]
}
B

Azure DevOps Service Connection UI Setup

1. Create Service Connection: Choose AWS.

2. Authentication Method: Select Workload Identity Federation.

3. Identity Provider Role (ARN): Enterprise-provided Role A ARN.

4. Role to Assume (ARN): Target Deployment Role B ARN.

5. Use OIDC: Ensure this checkbox is **Checked**.

The AWS Toolkit for Azure DevOps handles the dual-hop handshake (OIDC → STS → Target Role) entirely under the hood.

Common Issues

Credential Loading Failures

Ensure the service connection is authorized for all pipelines. If Role B cannot be assumed by Role A, verify the sts:AssumeRole permission on Role A's inline policy.
